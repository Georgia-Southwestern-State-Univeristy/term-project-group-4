
# ADR-001: User Authentication Strategy

## Status
**Proposed**

## Context
The Smart Packing Checklist Generator needs to allow users to save their trips, custom templates, and progress. This requires a persistent user identity. We evaluated two primary methods for managing these identities:

- **LDAP (Lightweight Directory Access Protocol)**: A traditional protocol for accessing and maintaining distributed directory information services.

- **OAuth 2.0 / OpenID Connect (OIDC)**: A modern framework that allows users to authenticate using existing accounts (e.g., Google, GitHub).

Given our project's scope, we need an authentication solution that provides high security with low maintenance overhead.

## Decision
We will use **OAuth 2.0 / OpenID Connect (OIDC)** (leveraging a provider like Google or Auth0) for user authentication instead of setting up an LDAP server.

## Rationale
- **Reduced Development Effort**: Setting up and maintaining an LDAP server is out of scope for a prototype focused on checklist logic. OAuth allows us to "outsource" the login UI and password recovery.

- **User Friction**: Our target users (travelers, parents, campers) are unlikely to have a custom LDAP identity. They are much more likely to have a Google or GitHub account, allowing for "One-Click" sign-in.

- **Security (Risk Mitigation)**: Storing user passwords in a database (as often required with LDAP integrations) increases our liability. By using OAuth, our application never sees or stores the user's password, reducing the risk of credential leaks.

## Consequences

**Positive:**
- We can focus our development time on the Rule-Based Checklist Logic rather than infrastructure management.

**Negative:**
- The application now depends on external service.
- We must handle "Token Expiration" logic to ensure users stay logged in while marking items as packed during their trip.
